# MEDIATOR — Testing Document

> 작성일: 2026-06-09

---

## 1. 테스트 철학

MEDIATOR의 테스트는 세 가지 질문에 답한다:

1. **코드 바꿨을 때 뭔가 망가졌는가?** → 단위 테스트
2. **API 요청/응답이 제대로 작동하는가?** → 통합 테스트
3. **실제 사용자 흐름이 정상인가?** → E2E 테스트 + 실사용 검증

음성 인식 품질은 테스트로 검증 불가 — **팀이 직접 써보는 것**이 유일한 검증 방법.

---

## 2. 실행 명령

```bash
npm run test:unit          # 단위 테스트 (빠름, ~1초)
npm run test:integration   # 통합 테스트 (~0.5초)
npm run test               # 단위 + 통합 전체
npm run test:coverage      # 커버리지 포함
npm run test:e2e           # E2E (로컬, 브라우저 필요)
```

E2E 처음 실행 시:
```bash
npx playwright install
npm run test:e2e
```

## 3. 단위 테스트 (17 → 24개)

**위치:** `src/__tests__/unit/`  
**도구:** Jest

### 3-1. 비속어 감지 (`profanity.test.ts`)

| 테스트 | 이유 |
|---|---|
| 정상 문장 → false | 오탐 방지 |
| 한국어 비속어 → true | 긴급 중재 트리거 |
| 영어 비속어 → true | 다국어 지원 |
| 대소문자 무시 | "FUCK" 같은 우회 차단 |

> 참고: 비속어 감지보다 **Claude가 맥락을 보고 판단**하는 게 더 중요. 정중하지만 반복적인 공격, "항상 이런 식이잖아요" 같은 표현은 키워드로 못 잡음.

### 3-2. 방 코드 생성 (`session.test.ts`)

| 테스트 | 이유 |
|---|---|
| 6자리 코드 | 짧으면 참가 불가 |
| I, O, 0, 1 미포함 | 헷갈려서 잘못 입력하면 참가 실패 |
| 100번 생성 → 90% 이상 다름 | 코드 충돌 가능성 낮음 확인 |
| createdAt 정렬 | 순서 틀리면 대화 흐름 뒤집힘 |

### 3-3. AI 프롬프트 빌더 (`prompts.test.ts`) ← 가장 중요

**왜 중요한가:** 프롬프트에 팀 이름/주제/대화가 빠지면 AI가 맥락 없이 generic한 중재만 함.

| 함수 | 핵심 테스트 케이스 |
|---|---|
| `buildSystemPrompt` | 팀 이름·주제 포함, 빈 값 시 undefined 방지 |
| `buildAutoPrompt` | 대화 내용 포함, SKIP 기준 명시, undefined 방지 |
| `buildAnalysisPrompt` | 3개 섹션 포함, 발화자 태깅 감지, 미수렴 이유 분석 지시 |

**제외한 테스트:**
- `buildUrgentCheckPrompt` — Claude 판단 영역, 테스트 의미 없음
- 비속어 필터 로직 — 핵심 가치가 아님

## 4. 통합 테스트 (5개)

**위치:** `src/__tests__/integration/`  
**도구:** Jest + Node 환경  
**대상:** `/api/chat` 라우트 (Anthropic SDK는 mock)

| 테스트 | 왜 필요한가 |
|---|---|
| 정상 요청 → 200 + content | 기본 동작 확인 |
| 허용되지 않은 origin → 403 | Claude API 키 보호 (외부 무단 호출 차단) |
| API 키 없음 → 500 | 환경변수 미설정 감지 |
| 100KB 초과 → 413 | 과도한 요청 차단 (긴 회의도 100KB면 충분) |
| `ai` role → `assistant` 변환 | Anthropic API 형식 준수 |

**페이로드 제한 근거:**  
한국어 1글자 ≈ 3바이트 → 100KB ≈ 33,000글자 ≈ 1,000개 발화.  
2시간 회의도 300-400개 발화 수준이므로 충분.

## 5. E2E 테스트 (6개)

**위치:** `e2e/`  
**도구:** Playwright (Chromium)  
**실행 환경:** 로컬 전용 (CI 환경은 BASE_URL 환경변수 사용)

| 시나리오 | 검증 내용 |
|---|---|
| 랜딩 페이지 로드 | 타이틀 + CTA 버튼 표시 |
| /ai 진입 | createOrJoin 화면, 이름 입력창 표시 |
| 이름 없이 생성 시도 | 오류 처리 (화면 전환 안 됨) |
| 참가하기 클릭 | 코드 입력창 + 취소 버튼 표시 |
| 존재하지 않는 방 코드 | 오류 메시지 표시 |
| /history 접근 | 페이지 정상 로드 |

### 음성 인식 E2E 미포함 이유

브라우저 테스트에서 마이크 시뮬레이션은 기술적으로 가능하지만:
- 실제 인식률 검증 불가 (가짜 오디오 스트림과 실제 음성은 다름)
- 환경마다 결과가 달라 테스트 신뢰도 낮음

→ **팀이 직접 실사용하는 것이 유일한 검증 방법**

## 6. 테스트 안 하는 것들 (의도적 제외)

| 항목 | 이유 |
|---|---|
| AI 응답 품질 | Claude 판단 영역, 프롬프트 품질로 간접 관리 |
| Firebase 실시간 동기화 | 실사용 환경에서만 의미 있음 |
| 음성 인식 정확도 | 브라우저 + 하드웨어 의존, 실사용 검증 |
| UI 스타일/레이아웃 | 자주 바뀌고 시각적 검증이 더 효과적 |

## 7. 새 테스트 추가하는 법

```bash
# 단위 테스트 추가
src/__tests__/unit/새파일.test.ts

# 통합 테스트 추가
src/__tests__/integration/새파일.test.ts
# 파일 상단에 반드시 추가:
# @jest-environment node

# E2E 추가
e2e/새시나리오.spec.ts
```

새 함수나 API 엔드포인트 만들 때 → 해당 테스트 같이 추가.
